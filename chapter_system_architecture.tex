\section{Overview}
% ─────────────────────────────────────────────────────────────
This chapter presents the complete architecture of the \textbf{Serverless Spark
Framework for Real-Time IoT Applications}. The framework integrates five functional
layers: (1)~IoT Edge with burst-prediction, (2)~Dual-lane MQTT+Kafka ingestion,
(3)~Two-Phase Hierarchical RL optimisation (TD3+PPO), (4)~OpenFaaS serverless
orchestration, and (5)~Dynamic Spark execution.

Unlike conventional single-layer Spark deployments, this architecture incorporates a
Two-Phase RL system and an OpenFaaS serverless function as the central decision engine,
enabling proactive, cost-aware, and latency-sensitive resource scaling. All modules are
fully containerised via Docker Compose for reproducibility and portability.

\noindent At runtime, IoT sensors emit telemetry every 2~s; the edge gateway computes
a burst-probability $P_{burst}$ and forwards it via MQTT. Bulk data flows through Kafka
while lightweight control signals travel over MQTT to avoid congestion. The TD3
Meta-Controller updates reward weights $(\alpha,\beta,\gamma)$ once per episode
($\sim$50~s), the PPO agent selects a scaling action every 5~s, and the OpenFaaS
function translates this into a concrete executor target for the Spark Driver REST API.
Step-level rewards close the PPO loop; episode statistics close the TD3 loop.

\section{Layer 2: Ingestion — Dual-Lane Kafka + MQTT}
\label{sec:layer2}
% ─────────────────────────────────────────────────────────────

An MQTT-to-Kafka bridge service forwards sensor payloads from MQTT topics into
corresponding Kafka topics for Spark consumption.

\section{Layer 4: OpenFaaS Serverless Orchestration}
\label{sec:layer4}
% ─────────────────────────────────────────────────────────────
Embedding the RL inference engine inside the Spark Driver creates tight coupling: any RL
model update requires redeploying the entire Spark application. OpenFaaS resolves this
by packaging the RL agent as a standalone HTTP microservice with three key advantages:
\textbf{Decoupling} (Spark calls the RL function via REST and is agnostic to its
implementation), \textbf{Scale-to-Zero} (the container terminates after 5 min of
inactivity, eliminating idle cost), and \textbf{Independent Updates} (a re-trained model
can be pushed without touching the Spark job).

The function is deployed via an OpenFaaS \texttt{stack.yml} descriptor with
\texttt{scale.min: 1} to keep one warm replica during active periods. It accepts a JSON
payload containing Spark telemetry (workload rate, CPU/memory utilisation, latency,
RL weights, burst probability) and returns a JSON scaling instruction (action type,
target executor count, reasoning string, confidence score).

% ─────────────────────────────────────────────────────────────
\section{Layer 5: Spark Execution Layer}
\label{sec:layer5}
% ─────────────────────────────────────────────────────────────
Apache Spark Structured Streaming performs windowed aggregations in 5-second
micro-batches with exactly-once semantics via checkpointing and automatic recovery after
executor or network failure. Dynamic executor allocation is driven by OpenFaaS scaling
commands issued through the Spark Driver REST API. The streaming job reads from all
three Kafka IoT topics, parses the JSON payloads, and invokes the OpenFaaS function via
the \texttt{foreachBatch} callback on each micro-batch to obtain the next scaling
decision.

\section{Pipeline Latency Analysis}
\label{sec:latency}

\[
  L_{total} = L_{mqtt} + L_{kafka} + L_{openfaas} + L_{spark} + L_{network}
\]
\[
  = 52 + 113 + 12 + 127 + 6 = 310\;\text{ms}
\]
Adding OpenFaaS introduces only 12~ms of additional latency (warm path) while
eliminating 100\% of SLA violations through proactive pre-warming — an extremely
favourable trade-off.

\section{Fault Tolerance and Resilience}
\label{sec:fault}
% ─────────────────────────────────────────────────────────────
A controlled failure test force-stopped the Kafka container mid-stream and restarted it
after 30 seconds. Spark automatically paused consumption, then resumed from the last
committed checkpoint offset — \textbf{zero messages were lost or duplicated}
(exactly-once semantics confirmed). The OpenFaaS function continued issuing decisions
from Redis-buffered metric snapshots throughout the outage. Adding executors dynamically
from 1 to 3 reduced micro-batch processing time by $\approx$50\%, validating the RL
agent's horizontal scaling capability.

\section{Summary}
% ─────────────────────────────────────────────────────────────
This chapter presented the complete system architecture in its final five-layer form.
The key architectural advancements are:
\begin{enumerate}
  \item A \textbf{dual-lane ingestion design} (Kafka + MQTT) isolating data traffic
        from RL control signals.
  \item A \textbf{Two-Phase Hierarchical RL Engine} (TD3 + PPO) separating strategic
        weight adaptation (50~s) from tactical scaling (5~s).
  \item \textbf{OpenFaaS serverless deployment} achieving scale-to-zero and clean
        component decoupling.
  \item \textbf{Proactive cold-start mitigation} using a 7-feature pattern extractor
        to trigger pre-warming before predicted bursts.
\end{enumerate}

\noindent Measured outcomes confirm $-$51.4\% latency, zero SLA violations, $-$95.6\%
idle cost, and $-$27.1\% total cost versus a traditional fixed-executor baseline.
Chapters~\ref{chap:rl_optimizer} and~\ref{chap:two_phase_rl_openfaas} detail the RL
formulations and training analyses.
