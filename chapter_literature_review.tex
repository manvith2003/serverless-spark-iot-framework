\section{Overview}
% ──────────────────────────────────────────────────────────────────────────────

The optimization of distributed data processing frameworks has evolved significantly over the past decade, transitioning from static cluster provisioning to dynamic, AI-driven serverless orchestration. This chapter systematically reviews the state-of-the-art literature across three intersecting domains that form the foundation of this thesis:
\begin{enumerate}
    \item \textbf{Auto-Scaling Mechanisms in Distributed Systems:} The transition from reactive threshold-based heuristics to predictive and serverless scaling.
    \item \textbf{Reinforcement Learning for Cloud Optimization:} The application of single-agent and Hierarchical Reinforcement Learning (HRL) to solve complex Markov Decision Processes (MDPs) in infrastructure management.
    \item \textbf{Edge-Cloud Collaboration:} The use of edge computing not merely for data offloading, but as an active telemetry source for predictive cluster sizing.
\end{enumerate}

\section{Auto-Scaling in Distributed Systems}
% ──────────────────────────────────────────────────────────────────────────────

Auto-scaling is the automated adjustment of computational resources to match workload demands, minimizing cost while honoring Service Level Agreements (SLAs).

\subsection{Threshold-Based Reactive Scaling}
The most prevalent industry approach is rule-based reactive scaling, utilized by Kubernetes Horizontal Pod Autoscalers (HPA) and AWS Auto Scaling groups. These systems rely on static upper and lower utilization thresholds:
\[ \text{ScaleOut if } CPU > T_{upper}, \quad \text{ScaleIn if } CPU < T_{lower} \]

While computationally simplistic and robust for stateless microservices, research by \cite{gandhi2018} and \cite{castro2020serverless} highlights critical failures when applying these heuristics to stateful topologies like Apache Spark. The primary bottleneck is the \textit{initiation overhead} or \textbf{cold start}. Because reactive systems wait for CPU saturation before triggering a scale-out, the cluster is actively overloaded during the 45--60 seconds required to boot JVM executors \cite{eismann2021serverless}. By the time resources are available, IoT SLAs are typically broken.

\subsection{Time-Series Predictive Scaling}
To mitigate the cold start problem, predictability models such as ARIMA, Prophet, and LSTMs have been introduced. Frameworks like Seer \cite{lu2016} forecast future load based on historical trends:
\[ N_{t+1} = \frac{\hat{L}_{t+1}}{C_{target}} \]
where $\hat{L}_{t+1}$ represents the predicted load and $C_{target}$ is the target per-node capacity. 

While predictive models natively handle diurnal traffic cycles, \cite{nestorov2024dexter} demonstrated that they perform poorly across highly non-stationary IoT workloads (e.g., sudden massive spikes caused by traffic accidents). Furthermore, pure predictive algorithms assume a strict linear relationship between CPU and throughput, entirely ignoring secondary bottlenecks like network I/O and shuffle disk contention \cite{pu2019shuffling}.

\section{Reinforcement Learning for Systems Optimization}
% ──────────────────────────────────────────────────────────────────────────────

Unlike predictive scaling which forecasts load, Reinforcement Learning (RL) learns a direct mapping from system states to scaling actions by treating orchestration as a Markov Decision Process (MDP).

\subsection{Single-Agent Approaches}
Early applications of RL in infrastructure were popularized by DeepMind’s reduction of Google Data Centre cooling costs \cite{evans2016}. In the context of big data, Mao et al. \cite{mao2016resource} utilized policy gradient algorithms for job scheduling, while more recent studies have applied Q-Learning and Deep Deterministic Policy Gradients (DDPG) to tune Spark executor counts and memory limits dynamically.

PPO (Proximal Policy Optimization) \cite{schulman2017ppo} has emerged as the standard for multi-objective cloud scheduling due to its stability and trust-region clipping, which prevents catastrophic policy collapse during training. However, the limitation of current single-agent RL research is the reliance on a \textbf{static reward function}. Authors typically combine cost and latency manually:
\[ R_t = - (\alpha \cdot L_{norm} + \beta \cdot C_{norm}) \]
If $\alpha$ and $\beta$ are fixed at training time, the agent cannot adapt to dynamic business environments (e.g., needing absolute low latency during the day, but prioritizing extreme cost savings at night).

\subsection{Hierarchical Reinforcement Learning (HRL)}
Hierarchical Reinforcement Learning decomposes complex MDPs into manageable sub-goals using a "Manager-Worker" paradigm. While widely researched in robotics for path planning, its application in cloud resource management remains emergent. 

Recent literature suggests HRL can solve the fixed-objective limitation. A Meta-Controller assesses macro-level business goals (e.g., SLA compliance rates over a 24-hour window) and outputs continuous adjustments to the reward weights $(\alpha, \beta, \gamma)$. The lower-level worker agent (e.g., PPO) treats these weights as observations and adjusts tactical scaling accordingly. This thesis builds directly upon this paradigm by utilizing Twin Delayed DDPG (TD3) as the continuous-action Meta-Controller \cite{zhou2024deep}.

\section{Edge-Cloud Collaboration and Telemetry}
% ──────────────────────────────────────────────────────────────────────────────

Edge Computing shifts processing to the network periphery. Traditionally, literature views the edge and cloud as mutually exclusive compute domains, fostering "task offloading" frameworks that mathematically decide whether an operation should execute locally or remotely.

\textbf{Identified Gap:} There is a distinct lack of literature treating the edge layer as an active \textit{Control Signal Generator} for serverless clouds \cite{symeonides2023sparkedge}. Because there is a network propagation delay between a sensor emitting data and that data overwhelming a cloud ingestion queue, the edge posesses a chronological advantage. 

This thesis exploits this gap. Rather than heavily processing data at the edge, we propose running lightweight temporal convolution layers (or pattern extractors) at the IoT gateway to generate a probabilistic \textit{Burst Signal}. Transmitted via a separate low-latency MQTT control lane, this signal alerts the cloud RL agent up to 45 seconds before the data tsunami hits Kafka, enabling proactive \texttt{PRE\_WARM} execution that entirely eliminates serverless cold-start SLA violations.

\subsection{Synthesis and Progression to Proposed Framework}

The literature confirms that while serverless abstraction solves physical provisioning, logical performance tuning remains deeply flawed under IoT traffic. Reactive scaling guarantees SLA breaches. Predictive scaling fails on non-stationary bursts. Single-agent RL proves mathematically superior but cannot shift business priorities without a complete retraining cycle.

By intersecting Hierarchical RL (for dynamic multi-objective scheduling), OpenFaaS (for decoupled scale-to-zero function execution), and Edge Telemetry (for cold-start mitigation), this thesis directly addresses the composite failures identified in the broader streaming analytics literature.
